Week 3 Review & Big Picture

Access control, polymorphism, inheritance, overriding, exceptions
›

Before diving into collections and interfaces, let's make sure the Week 3 foundations are solid. These concepts keep appearing throughout the rest of the course.

Key concepts from Week 3

  • Access control: private (only this class), protected (this class + subclasses), public (everyone)
  • Inheritance: extends lets a subclass reuse code from a superclass
  • Polymorphism: a superclass variable can hold any subclass object — the actual object determines which method runs
  • Overriding: a subclass provides its own version of a parent method using @Override
  • Substitutability (LSP): anywhere the superclass works, a subclass should too
  • Exceptions: throw creates an exception, try/catch handles it gracefully
✎ Quick check
When would you use protected instead of private for a field?
Use protected when subclasses need direct access to the field. Use private when only the class itself should access it (even subclasses must go through getters/setters). In general, prefer private and only use protected when subclasses genuinely need to read or modify the field.

Why We Need Collections

Arrays are fixed-size — real apps need dynamic data structures
›

So far, the only way to store multiple items in Java is an array. But arrays have a critical limitation: you must decide the size when you create them, and it can never change.

Real-World Analogy

Imagine a bookshelf with exactly 10 slots. If you have 11 books, you're stuck. If you only have 3 books, you're wasting 7 empty slots. A real library needs shelves that grow and shrink. That's what collections do for Java.

Think about a food delivery app like "ZipDish." You need to track:

  • Orders queue: new orders arrive constantly — you can't predict how many
  • Customer list: customers sign up and leave — the list changes
  • Menu items: restaurants add/remove dishes

With arrays, you'd have to guess a size. Too small? Crash. Too big? Wasted memory.

⚠️
Arrays are not wrong — they're limited. If you know the exact size at compile time (e.g., days of the week = 7), arrays are fine. But for dynamic data, you need something better.
✎ Quick check
What's wrong with using an array for a book catalogue where books are added and removed frequently?
Arrays have a fixed size set at creation. You can't add a book once the array is full (you'd get ArrayIndexOutOfBoundsException). You also can't easily remove a book from the middle without shifting all subsequent elements. A dynamic collection like ArrayList handles both automatically.

Building a Queue with Arrays

ZipDish Order class and array-based OrderManager
›

Let's build it step by step. First, create an Order class for the food delivery app:

public class Order {
    private int orderID;
    private String customer;
    private String restaurant;
    private String items;

    public Order(int orderID, String customer, String restaurant, String items) {
        this.orderID = orderID;
        this.customer = customer;
        this.restaurant = restaurant;
        this.items = items;
    }

    @Override
    public String toString() {
        return "Order #" + orderID + ": " + customer + " from " + restaurant;
    }
}

Now build an OrderManager using an array as a queue:

public class ArrayOrderManager {
    private Order[] orders;
    private int count = 0;

    public ArrayOrderManager(int capacity) {
        orders = new Order[capacity];  ← fixed size!
    }

    public void placeOrder(Order o) {
        orders[count] = o;  ← add to end
        count++;
    }

    public Order dispatchNext() {
        Order first = orders[0];  ← take from front
        for (int i = 0; i < count - 1; i++) {
            orders[i] = orders[i + 1];  ← shift everything left
        }
        count--;
        return first;
    }
}
💡
Queue = FIFO (First In, First Out). Orders are placed at the end and dispatched from the front — just like a real queue. The shifting in dispatchNext() is expensive but necessary with arrays.
✎ Quick check
Trace what happens when you place orders A, B, C and then dispatch one. What does the array look like after the dispatch?
After placing: [A, B, C], count=3. Dispatch returns A, then shifts: B moves to index 0, C moves to index 1. Array is now [B, C, _], count=2. The shifting is O(n) — every dispatch moves all remaining elements.
4

The Array Overflow Problem

What happens when the array is full?
›

The array-based OrderManager has a fatal flaw. If you create it with capacity 3 and try to place a 4th order:

ArrayOrderManager mgr = new ArrayOrderManager(3);
mgr.placeOrder(order1);  // OK
mgr.placeOrder(order2);  // OK
mgr.placeOrder(order3);  // OK
mgr.placeOrder(order4);  // 💥 ArrayIndexOutOfBoundsException!

The program crashes. The array has 3 slots, and orders[3] = o is out of bounds.

This is a fundamental problem: fixed-size arrays can't handle dynamic data. You could guess a very large capacity (e.g., 10,000), but that wastes memory. And you still might run out.

⚠️
This isn't just a theory problem. Real apps receive unpredictable amounts of data. A food delivery app can't crash because the lunch rush was bigger than expected.
5

Variable-Length Arrays

Resize by doubling — and let the garbage collector clean up
›

The manual solution: when the array is full, create a new array double the size, copy everything over, and reassign the reference.

public void placeOrder(Order o) {
    if (count == orders.length) {
        // Array is full — resize!
        Order[] bigger = new Order[orders.length * 2];
        for (int i = 0; i < orders.length; i++) {
            bigger[i] = orders[i];  ← copy all elements
        }
        orders = bigger;  ← point to new array
        // Old array is now unreferenced
    }
    orders[count] = o;
    count++;
}
👍
Java's garbage collector automatically frees memory for objects that no longer have any references. When orders is reassigned to the new array, the old array has zero references and gets cleaned up. In C, you'd need free() — forget it and you get a memory leak.
✎ Quick check
What happens to the old array after orders = bigger; is executed?
The old array no longer has any variable pointing to it (no references). Java's garbage collector detects this and automatically deallocates the memory. You don't have to do anything — this is one of Java's advantages over languages like C where you must manually free memory.
6

ArrayList — Java's Built-in Solution

Generics, automatic resizing, and clean API
›

All that manual resizing code? Java already has it built in: ArrayList. It's a resizable array that handles growth automatically.

Manual resizing (tedious)

Order[] orders = new Order[3];
int count = 0;
// Must handle resizing,
// shifting, count tracking…

ArrayList (clean)

ArrayList<Order> orders =
    new ArrayList<>();
// Resizing is automatic!
// No count tracking needed.

Key ArrayList methods

  • add(item) — append to end (auto-resizes if needed)
  • removeFirst() — remove and return the first element
  • size() — how many elements are in the list
  • isEmpty() — returns true if list has no elements
  • get(index) — access element at position

Here's the OrderManager rewritten with ArrayList:

import java.util.ArrayList;

public class ArrayListOrderManager {
    private ArrayList<Order> orders = new ArrayList<>();

    public void placeOrder(Order o) {
        orders.add(o);  ← that's it! No resize logic needed
    }

    public Order dispatchNext() {
        return orders.removeFirst();  ← removes and returns index 0
    }
}
💡
The <Order> part is called generics. It tells ArrayList what type of objects it holds. Without it, ArrayList would accept anything (Object), and you'd need type casts everywhere. Generics = type safety.
✎ Quick check
Rewrite the ArrayOrderManager's placeOrder and dispatchNext methods using ArrayList instead of an array.
private ArrayList<Order> orders = new ArrayList<>(); public void placeOrder(Order o) { orders.add(o); } public Order dispatchNext() { return orders.removeFirst(); } — All the resizing and shifting logic is gone. ArrayList handles it internally.

From Arrays to Interfaces

Two implementations, same public methods — time for a contract
›

Now we have two OrderManager classes: ArrayOrderManager and ArrayListOrderManager. Both have the same public methods:

  • placeOrder(Order o)
  • dispatchNext() returns an Order

Could we use inheritance? Make one extend the other? No — they don't share implementation code. One uses arrays internally, the other uses ArrayList. A superclass would have nothing in its body.

This is where interfaces come in. An interface says "any class that claims to be an OrderManager must provide these methods" — but it doesn't say how.

🔑
Inheritance = shared implementation. Interfaces = shared contract. When classes share behaviour (same methods) but not implementation (different internals), use an interface.
✎ Quick check
Why can't you use inheritance (a superclass) to unify ArrayOrderManager and ArrayListOrderManager?
Because they don't share implementation code. A superclass would have empty method bodies — there's no "default" version of placeOrder() that both could inherit. They agree on what methods to have (the contract) but not how to implement them. That's exactly what an interface is for.
8

Interfaces as Contracts

Declare method signatures without implementation — classes "sign the contract"
›

An interface declares method signatures (name, parameters, return type) but provides no implementation — no method bodies, no fields, no constructors.

public interface OrderManager {
    void placeOrder(Order o);     ← no body, just a semicolon
    Order dispatchNext();          ← no body, just a semicolon
}

A class "signs the contract" using implements:

public class ArrayOrderManager implements OrderManager {
    // MUST provide placeOrder() and dispatchNext()
    // or the compiler will refuse to compile

    @Override
    public void placeOrder(Order o) {
        // ... array-based implementation ...
    }

    @Override
    public Order dispatchNext() {
        // ... array-based implementation ...
    }
}

public class ArrayListOrderManager implements OrderManager {
    @Override
    public void placeOrder(Order o) {
        // ... ArrayList-based implementation ...
    }

    @Override
    public Order dispatchNext() {
        // ... ArrayList-based implementation ...
    }
}
⚠️
The compiler enforces the contract. If you write implements OrderManager but forget to implement dispatchNext(), the code won't compile. The interface is a guarantee.

Key syntax differences

  • interface instead of class in the declaration
  • Methods have no body — just ; after the signature
  • Classes use implements (not extends) to adopt an interface
  • The implementing class must provide all methods from the interface
✎ Quick check
Write an OrderManager interface with two methods: placeOrder(Order o) and dispatchNext() that returns an Order.
public interface OrderManager { void placeOrder(Order o); Order dispatchNext(); } — Note: no method bodies, no curly braces on the methods. Just the signatures followed by semicolons.
9

Polymorphism via Interfaces

Declare the interface type, swap implementations freely
›

Just like with inheritance, you can declare a variable as the interface type and assign any implementing class to it:

// Declare as the interface type
OrderManager manager = new ArrayListOrderManager();

// Use it — client code doesn't know (or care) which implementation
manager.placeOrder(order1);
manager.placeOrder(order2);
Order next = manager.dispatchNext();

// Want to swap implementations? Change ONE line:
OrderManager manager = new ArrayOrderManager(100);
// All other code stays exactly the same!

Key advantages of interfaces over inheritance

  • Multiple interfaces: a class can implements many interfaces (Java only allows one extends)
  • Loose coupling: client code depends on the interface, not the implementation
  • Swap freely: change implementations by editing one line
🔑
"Program to an interface, not an implementation." This is one of the most important principles in software design. Your code should depend on what something can do (interface), not how it does it (class).
✎ Quick check
Why can't you use inheritance (extends) to achieve the same flexibility as interfaces here? Give two reasons.
(1) No shared implementation: a superclass would have empty method bodies — there's no default code to inherit. (2) Single inheritance: Java only allows one extends, so if ArrayListOrderManager already extends another class, it can't also extend OrderManager. Interfaces solve both: no implementation needed, and a class can implement multiple interfaces.

Tutorial Preview: Lists & Maps

ArrayList, LinkedList, HashMap, TreeMap, generics, and the == trap
›

Tutorial 4 dives deeper into Java's collections framework. Here's a preview of what you'll encounter:

Lists: ArrayList vs LinkedList

  • ArrayList — fast random access (get(i)), slow insert/remove in middle
  • LinkedList — fast insert/remove anywhere, slow random access
  • Both implement the List interface — you can swap them!

Maps: like Python dictionaries

// Python dictionary
// prices = {"burger": 12.50, "pizza": 15.00}

// Java HashMap — same idea
HashMap<String, Double> prices = new HashMap<>();
prices.put("burger", 12.50);
prices.put("pizza", 15.00);
double p = prices.get("burger");  // 12.50

Critical gotcha: wrapper classes and ==

Generics only work with object types, not primitives. Use wrapper classes:

  • int → Integer, double → Double, boolean → Boolean
  • ArrayList<int> is illegal — use ArrayList<Integer>
Integer a = 1000;
Integer b = 1000;
System.out.println(a == b);       // false! (compares references)
System.out.println(a.equals(b));  // true  (compares values)
⚠️
== on Integer objects compares references (memory addresses), not values. Two separate Integer objects with value 1000 are at different addresses, so == returns false. Always use .equals() to compare wrapper objects. (Note: Java caches integers -128 to 127, so == works for small numbers — but don't rely on this!)

Week 4 in one sentence

Arrays are fixed-size, so Java provides ArrayList (auto-resizing with generics); when multiple classes share the same method signatures but not implementation, use an interface to define the contract, enabling polymorphism that lets you swap implementations by changing a single line of code.

✎ Quick check
Why does Integer a = 1000; Integer b = 1000; System.out.println(a == b); print false?
== on objects compares references (memory addresses), not values. a and b are two separate Integer objects that happen to both contain 1000, but they're stored at different memory locations. Use a.equals(b) to compare the actual integer values.
📝 Week 4 Self-Test
10 exam-style questions. Try to answer each one before revealing. These cover the full study guide.
Q1Remembering

What is the difference between an array and an ArrayList?

An array has a fixed size set at creation and cannot grow or shrink. An ArrayList automatically resizes as elements are added or removed. Arrays use [] syntax; ArrayList uses methods like add(), get(), size(). Arrays can hold primitives; ArrayList requires object types (wrapper classes).
Q2Understanding

Explain why Java's garbage collector matters when resizing arrays.

When you resize by creating a new larger array and reassigning the reference, the old array has no references left. Java's garbage collector automatically detects unreferenced objects and frees their memory. Without a garbage collector (like in C), the programmer must manually call free() — forgetting to do so causes a memory leak where unused memory is never released.
Q3Applying

Write the code to create an ArrayList of Strings and add 3 items: "Alice", "Bob", "Charlie".

ArrayList<String> names = new ArrayList<>(); names.add("Alice"); names.add("Bob"); names.add("Charlie");
Q4Applying

Given an interface Sortable with one method void sort(), write a class BubbleSorter that implements it.

public class BubbleSorter implements Sortable { @Override public void sort() { // bubble sort implementation here } } — The class must use implements (not extends) and must provide a body for every method declared in the interface.
Q5Analysing

A student writes OrderManager om = new OrderManager(); where OrderManager is an interface. What happens and why?

Compiler error. You cannot instantiate an interface — interfaces have no constructors and no implementation. You can only declare a variable as the interface type and assign an implementing class: OrderManager om = new ArrayListOrderManager();. The interface is a contract, not a blueprint.
Q6Understanding

Explain the difference between inheritance and interfaces for achieving polymorphism.

Inheritance provides polymorphism through shared implementation — the superclass has actual code that subclasses reuse or override. Interfaces provide polymorphism through shared contracts — no code, just method signatures. Inheritance is "is-a" with code sharing; interfaces are "can-do" contracts. A class can implement multiple interfaces but can only extend one class.
Q7Applying

Trace the execution of dispatching from an array-based queue that contains orders [A, B, C]. Show the array state after dispatch.

1. first = orders[0] → saves A. 2. Shift loop: orders[0] = orders[1] (B moves to 0), orders[1] = orders[2] (C moves to 1). 3. count-- → count becomes 2. 4. Return A. Final array: [B, C, _] with count=2.
Q8Analysing

Why does Integer a = 1000; Integer b = 1000; a == b return false?

== on objects compares references (memory addresses). a and b are two separate Integer objects at different memory locations, even though both hold value 1000. Use a.equals(b) for value comparison. (Java caches Integer values -128 to 127, so small values might give true with == — but never rely on this.)
Q9Applying

Convert this array code to use ArrayList: String[] names = new String[3]; names[0] = "Alice";

ArrayList<String> names = new ArrayList<>(); names.add("Alice"); — No size needed at creation. Use add() instead of index assignment. To access: names.get(0) instead of names[0].
Q10Analysing

A class implements both Printable and Loggable interfaces. Both declare void output(). Is this a problem? Explain.

No, it's not a problem. The class provides one implementation of output(), which satisfies both interfaces simultaneously. Since interfaces only declare signatures (no bodies), there's no conflict — the single method body fulfils both contracts. This is different from the "diamond problem" in multiple inheritance (which Java avoids by only allowing single class inheritance).
