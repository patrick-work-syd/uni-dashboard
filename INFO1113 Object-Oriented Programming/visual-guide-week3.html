What is Inheritance?

One class borrows from another — code reuse through hierarchy
›

In Week 1, you created separate classes like Person and Car. Each had its own fields and methods. But sometimes you notice classes share behaviour — they have things in common.

Real-World Analogy

Think of a university: there are People (name, date of birth, address). Some people are Students (who are also people, but have extra fields like "student ID" and "GPA"). Some are Staff (who are also people, but have "employee ID" and "salary"). All of them share the basic person data. Why write "name, address, etc." twice?

Inheritance is how Java solves this: Student extends Person means "Student is a special kind of Person." Student automatically gets all of Person's fields and methods, then adds its own.

The parent class is called the superclass. The child class is called the subclass. The subclass inherits (gets for free) all public fields and methods from the superclass, then adds its own.

👍
This is DRY (Don't Repeat Yourself). Instead of writing name, age, getName() in both Student and Staff, you write it once in Person.
✎ Quick check
If Student extends Person, list 4 things a Student object automatically has, without you writing them in the Student class.
name field, age field, getName() method, getAge() method. These come from Person. The Student class only needs to define studentID, gpa, and getGPA().

The extends Keyword

How to write inheritance in Java syntax
›

Here's the syntax. In Java, you use the keyword extends to say "this class is a subclass of that class."

public class Person {
    private String name;
    private int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public String getName() { return this.name; }
    public int getAge() { return this.age; }
}

public class Dog extends Person {  ← Dog is a subclass of Person
    private String breed;

    public Dog(String name, int age, String breed) {
        super(name, age);  ← Call parent's constructor
        this.breed = breed;
    }

    public String getBreed() { return this.breed; }
}

Key points

  • extends Person means Dog inherits from Person
  • super(name, age) calls the parent's constructor to set up name and age
  • Dog doesn't redeclare name or age — they come from Person
  • Dog is a private, not public superclass member — subclasses inherit them differently
⚠️
private fields are NOT inherited. If Person's name and age are private, Dog can't access them directly. You must call super(name, age) in the constructor to let the parent store them. Or use public/protected fields (more on Step 5).
✎ Quick check
Write the first line of a class called Cat that extends Animal. Then write a constructor that calls super() to pass a name.
public class Cat extends Animal { public Cat(String name) { super(name); ... } }

Method Overriding with @Override

When a subclass wants its own version of a parent method
›

Sometimes the subclass needs to do something differently than the parent. That's called overriding a method — you write a new version in the subclass.

Before override — both same

Animal dog = new Dog();
dog.speak();
// Output: "Some sound"
// (uses parent's method)

After override — subclass version runs

Animal dog = new Dog();
dog.speak();
// Output: "Woof!"
// (uses Dog's own method)

Here's how to override:

public class Animal {
    public String speak() {
        return "Some sound";
    }
}

public class Dog extends Animal {
    @Override  ← Tells Java this is intentional
    public String speak() {
        return "Woof!";
    }
}

The @Override annotation (the @ symbol) tells Java: "I'm intentionally overriding a parent method." If you misspell the method name, Java throws a compiler error instead of silently creating a new method. It's a safety net.

💡
@Override is not required, but it's best practice. Always use it when overriding a parent method — the compiler catches typos.
✎ Quick check
Write a Dog class that extends Animal. Override the speak() method to return "Woof!" instead of "Some sound".
public class Dog extends Animal { @Override public String speak() { return "Woof!"; } }

toString(): Making Objects Printable

Control how your objects look when printed
›

Every class in Java automatically inherits from a class called Object (even if you don't explicitly extend it). Object has a method called toString() — it's what gets called when you print an object.

By default, it's useless:

Without toString() override

Student s = new Student("Will", 3.8);
System.out.println(s);
// Output: Student@7852e922
// (memory address — useless!)

With toString() override

Student s = new Student("Will", 3.8);
System.out.println(s);
// Output: Student[name=Will, gpa=3.8]
// (readable!)

Override toString() to return a readable String:

public class Student {
    private String name;
    private double gpa;

    // ... constructor ...

    @Override
    public String toString() {
        return "Student[name=" + name + ", gpa=" + gpa + "]";
    }
}
👍
toString() is a form of @Override. You're overriding Object's default toString() with your own version. Use @Override before it.
✎ Quick check
Write a toString() method for a Book class with fields title (String) and pages (int). Make it return "Book[title=..., pages=...]"
@Override public String toString() { return "Book[title=" + title + ", pages=" + pages + "]"; }

The protected Access Modifier

Three access levels: public, private, protected
›

In Week 1, you learned public (anyone can access) and private (only this class). Now there's a third: protected — this class AND its subclasses.

Use protected when a field is private implementation detail for non-subclasses, but subclasses need to access or modify it:

public class Animal {
    protected String name;  ← Subclasses can access, others can't
    private int age;      ← Only Animal can access

    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }
}

public class Dog extends Animal {
    public String getFullName() {
        return this.name + " the Dog";  ← Can access protected name
    }
}
⚠️
Be careful with protected. It exposes internals to subclasses. Generally: fields are private, methods are public, and use protected rarely.
✎ Quick check
If Animal has a protected field called "name" and a private field called "id", can a Dog subclass access both, either, or neither?
Dog can access name (protected) but not id (private). Protected means subclasses have access; private means only Animal can access it.

Polymorphism: One Variable, Many Forms

A superclass variable can hold any subclass object
›

Polymorphism means "many forms." Here's the key idea: a variable of type Animal can hold a Dog object, a Cat object, or any subclass of Animal.

Without inheritance (separate variables)

Dog dog = new Dog();
Cat cat = new Cat();
Bird bird = new Bird();

dog.speak();   // Woof!
cat.speak();   // Meow!
bird.speak();  // Chirp!

With polymorphism (one variable type)

Animal dog = new Dog();
Animal cat = new Cat();
Animal bird = new Bird();

dog.speak();   // Woof!
cat.speak();   // Meow!
bird.speak();  // Chirp!

The right side (which object is created) determines the behaviour. The left side (the variable type) just says "this is-a type of Animal."

speak(), Java uses the actual object's method (Dog), not the variable type (Animal).

This is powerful: write a function that accepts Animal, and it works with Dog, Cat, Bird, or any subclass:

public void makeSound(Animal a) {
    System.out.println(a.speak());  ← Works for ANY subclass of Animal
}
🔑
This is the power of inheritance: write functions that work with the parent type, and they automatically work with all subclasses — now and in the future.
✎ Quick check
You have Animal cat = new Cat(); and call cat.speak(). Does it run Animal's speak() or Cat's speak()? Why?
It runs Cat's speak(). The actual object is a Cat, so Cat's overridden method is called. The variable type (Animal) is just the declared type — the actual object determines the behaviour.

Liskov Substitution Principle

A design rule: subclasses should be drop-in replacements
›

Here's a simple principle: if your code works with a superclass, it must work with any subclass without breaking.

Named after Barbara Liskov, it's basically common sense about inheritance. If you write a method that expects an Animal, you should be able to pass a Dog, Cat, or Bird without the function breaking.

Real-World Analogy

If you're a chef and a recipe says "use a container," you should be able to use any container (pot, bowl, pan) without breaking the recipe. A subclass is like a specialized container — it should work wherever the general container works.

Violating LSP (Bad Design)

public class Bird extends Animal {
    public void fly() { System.out.println("Flying!"); }
}

public class Penguin extends Bird {
    @Override
    public void fly() {
        throw new UnsupportedOperationException("Penguins don't fly!");  ← BREAKS LSP!
    }
}

This violates LSP because code expecting a Bird and calling fly() breaks when given a Penguin. Better design: don't make Penguin extend Bird. Or make Bird more general.

💡
In practice: if you catch yourself throwing "unsupported" exceptions in a subclass method, your inheritance hierarchy might be wrong. Rethink the design.
✎ Quick check
If a method expects a Vehicle and calls start(), and you pass a Car (subclass of Vehicle), should start() work? Explain in terms of LSP.
Yes, absolutely. LSP says that anywhere the code accepts a Vehicle, a Car should work without breaking. If Car's start() breaks the contract, the inheritance design is wrong.

Exceptions: When Things Go Wrong

Java's try-catch is like Python's try-except
›

In Python (lessons 1-8), you used try-except to catch errors. Java has try-catch — it's the same idea.

Python try-except

try:
    x = 10 / 0
except ZeroDivisionError:
    print("Can't divide by zero!")

Java try-catch

try {
    int x = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("Can't divide by zero!");
}

When an error happens, an exception is thrown. Your try-catch catches it and handles it gracefully instead of crashing.

Common exceptions you'll see

  • NullPointerException — you tried to use a null object (e.g., String s = null; s.length();)
  • ArrayIndexOutOfBoundsException — you accessed array[i] where i is too big or negative
  • FileNotFoundException — you tried to open a file that doesn't exist
  • ArithmeticException — you divided by zero
👍
Without try-catch, the program crashes. With it, you catch the error and decide what to do — print a message, retry, or exit gracefully.
✎ Quick check
What's the difference between a try-catch block running successfully (no error) vs catching an exception? What happens in each case?
If no error: the try block completes, and the catch block is skipped. If an error occurs: the try block stops immediately, and the catch block runs to handle it. In both cases, the program continues after the try-catch (doesn't crash).

Defensive Programming

Checking for bad input BEFORE it crashes
›

The best exception is the one that never happens. Defensive programming means checking for bad data before it causes an error.

Without defensive checks (crashes)

public void setAge(int age) {
    this.age = age;  ← What if age is -5?
}

With defensive checks (safe)

public void setAge(int age) {
    if (age < 0) {
        throw new IllegalArgumentException(
            "Age cannot be negative!"
        );
    }
    this.age = age;
}

Common defensive checks

  • Check if a parameter is null: if (name == null) throw new IllegalArgumentException(...);
  • Check if a number is in valid range: if (age < 0 || age > 150) throw ...;
  • Check if an array index is valid: if (index < 0 || index >= arr.length) throw ...;
  • Check if a String is empty: if (name.trim().isEmpty()) throw ...;
💡
Fail fast, fail loud. Throw an exception with a clear message as soon as you detect bad data. Don't let bad data silently propagate — it makes bugs much harder to find.
✎ Quick check
Write two defensive checks in a constructor: (1) name cannot be null, (2) gpa must be between 0.0 and 4.0. Use throw new IllegalArgumentException(message).
if (name == null) throw new IllegalArgumentException("Name cannot be null"); if (gpa < 0.0 || gpa > 4.0) throw new IllegalArgumentException("GPA must be 0.0-4.0");

File I/O: Reading Files

Scanner to read line by line, handling FileNotFoundException
›

In Python, you use open(). In Java, you use Scanner with a File. You must handle FileNotFoundException with try-catch.

Python file reading

with open('data.txt') as f:
    for line in f:
        print(line.strip())

Java file reading

try {
    Scanner sc = new Scanner(new File("data.txt"));
    while (sc.hasNextLine()) {
        System.out.println(sc.nextLine());
    }
    sc.close();
} catch (FileNotFoundException e) {
    System.out.println("File not found!");
}

Key points:

  • new Scanner(new File(filename)) opens the file
  • hasNextLine() checks if there's another line to read
  • nextLine() reads the next line (without the newline character)
  • sc.close() closes the file — important for cleanup
  • FileNotFoundException is thrown if the file doesn't exist — must catch it
⚠️
Always close the file. After reading, call sc.close() to release the file. Or use try-with-resources (Java 7+): try (Scanner sc = new Scanner(...)) { ... } — it auto-closes.
✎ Quick check
Write code to read "students.txt" line by line using Scanner and print each line, with proper try-catch for FileNotFoundException.
try { Scanner sc = new Scanner(new File("students.txt")); while(sc.hasNextLine()) System.out.println(sc.nextLine()); sc.close(); } catch(FileNotFoundException e) { System.out.println("File not found"); }

File I/O: Writing Files

PrintWriter to write to a file, and why you must close it
›

To write to a file, use PrintWriter. It's similar to System.out.println() but writes to a file instead of the console.

Python file writing

with open('output.txt', 'w') as f:
    f.write('Hello, file!')

Java file writing

try {
    PrintWriter pw = new PrintWriter(new File("output.txt"));
    pw.println("Hello, file!");
    pw.println("Second line");
    pw.close();
} catch (FileNotFoundException e) {
    System.out.println("Cannot write to file!");
}

Key points:

  • new PrintWriter(new File(filename)) opens the file for writing
  • pw.println(text) writes a line (adds newline)
  • pw.print(text) writes without adding newline
  • pw.close() must be called — without it, the file won't actually save!
  • If file exists, it's overwritten. If not, it's created
⚠️
CRITICAL: You MUST call close(). PrintWriter buffers output — data doesn't actually write to the file until you close. If you forget, your file will be empty.
✎ Quick check
Write code to create a file "results.txt" and write two lines: "Result: Pass" and "Score: 95". Don't forget close()!
try { PrintWriter pw = new PrintWriter(new File("results.txt")); pw.println("Result: Pass"); pw.println("Score: 95"); pw.close(); } catch(FileNotFoundException e) { System.out.println("Cannot write"); }

Putting It All Together: Pet Class Hierarchy

Inheritance, @Override, exceptions, file I/O in one program
›

Here's a complete example using inheritance (Dog extends Pet), method overriding, toString(), defensive programming, and file I/O:

Pet.java — superclass

public class Pet {
    protected String name;
    protected int age;

    public Pet(String name, int age) {
        if (name == null) throw new IllegalArgumentException("Name cannot be null");
        this.name = name;
        this.age = age;
    }

    public String speak() { return "Some sound"; }

    @Override
    public String toString() { return "Pet[" + name + "," + age + "]"; }
}

Dog.java — subclass, overrides speak()

public class Dog extends Pet {
    private String breed;

    public Dog(String name, int age, String breed) {
        super(name, age);  ← Call parent constructor
        this.breed = breed;
    }

    @Override
    public String speak() { return "Woof! I'm a " + breed; }

    @Override
    public String toString() { return "Dog[" + name + "," + breed + "]"; }
}

PetMain.java — uses inheritance and file I/O

import java.io.*;
import java.util.*;

public class PetMain {
    public static void main(String[] args) {
        Pet generic = new Pet("Fluffy", 3);
        Pet dog = new Dog("Rex", 5, "Labrador");

        System.out.println(generic.speak());  // Some sound
        System.out.println(dog.speak());      // Woof! I'm a Labrador

        ← Polymorphism: both variables are Pet type, but different behaviour

        try {
            PrintWriter pw = new PrintWriter(new File("pets.txt"));
            pw.println(generic.toString());
            pw.println(dog.toString());
            pw.close();
        } catch (FileNotFoundException e) {
            System.out.println("Error writing pets");
        }
    }
}
🔑
What this program shows: Dog extends Pet (inheritance), speaks differently (override), has a readable toString(), uses defensive checks (null check), and writes to a file with proper error handling.
✎ Quick check
In the PetMain code, why can you assign a Dog object to a Pet variable? What happens when you call speak() on that variable?
Because Dog extends Pet (is a subclass), a Dog can be assigned to a Pet reference. When you call speak(), Java uses Dog's overridden version (polymorphism), so it prints "Woof! I'm a Labrador" not "Some sound".

Week 3 in one sentence

Java organises related classes into inheritance hierarchies (superclass/subclass with extends), lets subclasses override parent methods with @Override, supports polymorphism (a superclass variable holding a subclass object), handles errors with try-catch, prevents bugs with defensive checks, and reads/writes files with Scanner and PrintWriter.

Q2Understanding

Explain the difference between overriding a method and overloading it.

Overriding: subclass replaces a parent method (same name, same parameters, different body). Overloading: same class has multiple methods with the same name but different parameters. Override is about inheritance; overload is about convenience in one class.
Q3Understanding

What does the @Override annotation do? What happens if you misspell the method name?

@Override tells the compiler "I intend to override a parent method." If you misspell the method name, the compiler throws an error instead of silently creating a new method — it's a safety net.

Q5Understanding

Explain what polymorphism means using the Animal/Dog/Cat example.

Polymorphism means a variable of type Animal can hold a Dog or a Cat. When you call animal.speak(), Java calls the actual object's version (Dog's or Cat's), not Animal's. The variable type is the "shape" but the object determines the behaviour.
Q6Understanding

What is the Liskov Substitution Principle in simple terms?

If your code works with a superclass, it should work with any subclass without breaking. Example: if a method expects an Animal, passing a Dog should work perfectly. If Dog breaks that contract, the inheritance design is wrong.
